# Story 1.1: Monorepo Setup — Turborepo + Workspaces Configuration

## Story Statement

**Como** desenvolvedor do MaxCapital,
**Eu quero** que o projeto esteja configurado como um monorepo funcional com Turborepo e npm workspaces,
**Para que** os pacotes `@maxcapital/shared`, `@maxcapital/api` e `@maxcapital/web` possam ser desenvolvidos, buildados e testados de forma integrada.

## Context

O projeto atualmente é uma SPA React rodando em `src/` com mock data. Já existe um `turbo.json` mínimo e um `packages/shared/` scaffoldado (estrutura de diretórios + package.json), mas o root `package.json` não tem `workspaces` configurado e os pacotes `api` e `web` não existem.

**Estado atual:**
- `package.json` root: sem `workspaces`, nome genérico (`vite_react_shadcn_ts`)
- `turbo.json`: pipeline básico existe mas sem workspace awareness
- `tsconfig.json` root: referencia `tsconfig.app.json` e `tsconfig.node.json`, sem workspace references
- `packages/shared/`: estrutura completa (19 arquivos), `@maxcapital/shared` configurado
- `packages/api/`: NAO EXISTE
- `packages/web/`: NAO EXISTE
- `src/`: aplicacao React completa (frontend funcional com mock data)

## Dev Notes

### Decisao de Estrategia: Mover vs Wrapper
A arquitetura define `packages/web/src/` como destino final do frontend. Porem, mover 1590 arquivos nesta story quebraria o escopo atomico. A estrategia desta story e:
1. Criar `packages/web/` como **wrapper** apontando para `src/` existente
2. Story 1.4 fara o **move fisico** de `src/` -> `packages/web/src/`

Isso permite que o monorepo funcione imediatamente sem risco de quebra.

### Conflito de Types (snake_case vs camelCase)
Existem dois sistemas de tipos conflitantes:
- `src/types/supabase.ts` — snake_case (match database)
- `src/types/index.ts` — camelCase (JS convention)

Isso NAO e escopo desta story (sera resolvido na Story 1.2 durante a extracao para `packages/shared`).
[Source: src/types/supabase.ts, src/types/index.ts]

### packages/shared Existente
O pacote `packages/shared/` ja foi scaffoldado com:
- 19 arquivos de tipo (auth, operations, companies, etc.)
- Constants (pipeline, segments, roles)
- Validators (auth, common, companies)
- package.json com nome `@maxcapital/shared`

O conteudo dos arquivos pode estar vazio/stub — sera populado na Story 1.2.
[Source: packages/shared/package.json]

### Vite Config
O `vite.config.ts` atual esta na raiz com alias `@/ → ./src`. Quando `packages/web/` for criado como wrapper, o pacote precisara de seu proprio `vite.config.ts` apontando para o `src/` da raiz via alias relativo (`../../src`). Este arquivo DEVE ser criado como parte da Task 4 (adicionado ao File List).

Vite config minimo para `packages/web/vite.config.ts`:
```typescript
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react-swc';
import path from 'path';

export default defineConfig({
  plugins: [react()],
  resolve: {
    alias: { '@': path.resolve(__dirname, '../../src') },
  },
});
```
[Source: vite.config.ts]

### packages/shared — Fixes Necessarios
Identificados durante validacao @po:
1. `packages/shared/tsconfig.json` nao tem `composite: true` — NECESSARIO para TypeScript project references (Task 5.2 depende disso)
2. `packages/shared/package.json` nao tem scripts `test` e `lint` — `turbo run test` e `turbo run lint` falharao sem eles

O @dev deve incluir esses fixes como parte da Task 5 ou como subtask adicional.

### Task 4.1 — Copiar, NAO Mover Dependencias
A instrucao "Mover dependencias de React/Vite/UI" deve ser interpretada como **COPIAR** as deps para `packages/web/package.json`, mantendo-as tambem no root `package.json`. Remover do root quebraria AC-10 (frontend continua funcionando via `npm run dev` na raiz). O cleanup definitivo ocorre na Story 1.4.

### turbo.json — Ja Esta Correto
